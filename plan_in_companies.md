How companies plan these systems
The first thing a senior engineer does is separate concerns ruthlessly. They think in three orthogonal axes: data flow (how a document becomes an answer), control flow (how a query moves through the system), and feedback flow (how eval scores drive improvements). If you can't change your chunking strategy without touching your LLM call, your architecture has failed.
What tools companies actually use
For the ingestion layer, the battle-tested stack is Docling or Unstructured for PDF parsing (they handle tables, headers, and multi-column layouts, which PyMuPDF alone won't), followed by LangChain's RecursiveCharacterTextSplitter or semantic chunking via sentence embeddings. Chunk size and overlap are treated as tunable hyperparameters — not constants — because they vary wildly by document type.
For the vector store, Qdrant and Weaviate are the current industry favorites for self-hosted. Pinecone for fully managed. The key is that all of them support hybrid search natively now (dense + sparse in one query), so you don't need to maintain two separate indexes and merge manually.
For the reranker, FlashRank (which you're already using) and Cohere Rerank are both solid. The mental model is: retrieval is cheap and broad (fetch top-50), reranking is expensive and precise (cut to top-5). Never send all 50 chunks to the LLM.
For observability, LangSmith or Langfuse are non-negotiable in production. Every chunk retrieved, every reranker score, every LLM prompt — it all gets traced. Without this, debugging a wrong answer is pure guesswork.
What they keep in mind (and most tutorials skip)
The things that bite teams hardest in production are: (1) PDF parsing quality — garbage in, garbage out. A scanned PDF with no OCR will produce semantically useless chunks. (2) Chunk boundary problems — splitting mid-sentence destroys context. (3) Query-document mismatch — a user asks "what's the revenue?" but the doc says "total earnings were..." — this is where HyDE (Hypothetical Document Embeddings) or query expansion helps. (4) Stale indexes — you need a strategy for re-ingesting when source documents are updated.
Your concrete next steps, in order
Since you've already refactored to a class-based architecture with YAML configs, here's the sequence that makes sense:
First, harden your ingestion. Add a document hash check so re-running doesn't duplicate embeddings. Validate chunk quality by logging chunk length distribution — if your p5 chunk is 12 tokens, your splitter is broken.
Second, wire up Langfuse (open source, self-hostable) before adding more features. You want observability before you build the eval loop, not after. Every pipeline run should emit a trace with retrieval scores, reranker scores, and the final prompt.
Third, implement the RAGAS eval as a CLI command (python pipeline.py eval --dataset test_cases.json). Generate your test cases semi-automatically: run 20 queries through the pipeline, have the LLM generate ground-truth answers, freeze those as your regression suite.
Fourth, add a CI gate — a GitHub Actions step that runs your eval suite on every PR and fails if faithfulness drops below a threshold. This is the single thing that separates "research project" from "production system."
